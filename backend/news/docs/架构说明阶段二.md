# 架构说明阶段二
> 本文档基于 [架构说明阶段一](/Users/gao/Desktop/Flex-Practicum-Project-2026/backend/news/docs/架构说明阶段一.md) 继续设计。  
> Phase 1 已经解决“先抓到、先看见、先可解释”；Phase 2 要解决“更干净、更稳定、更像正式新闻产品”。

---
## 一、Phase 2 的目标
Phase 2 不推翻 Phase 1，只在现有主链路上加一层“质量提升层”：
1. 先去重，再写缓存
2. 给 `Top News` 增加正式候选和排序逻辑
3. 把 `Trending` 从占位区块升级成真正的话题区块
4. 把 `sec_filing_rss` 从新闻主流剥离到独立 SEC 视图
5. 允许少量专题增强 query
6. 明确哪些调试字段退出正式 UI

进入 Phase 2 时，默认沿用这些 Phase 1 基线：
- Brave 仍是主来源
- RSS 仍是补充来源
- `company_queries` 仍固定为 4 个 `intent`
- preview 与正式模板规则已经统一
- 新增公司默认只有 Brave，没有自动 RSS

---
## 二、底层架构
Phase 2 推荐把主链路升级成下面这一版：

```text
query_engine
  -> fetcher
  -> pipeline
  -> cache
  -> ranking
  -> routes
```

其中职责建议固定如下：

| 层 | 文件 | Phase 2 职责 |
|---|---|---|
| 查询层 | `query_engine.py` | 继续生成基础模板；按需增加专题增强模板 |
| 抓取层 | `fetcher.py` | 继续抓 Brave + RSS；不在这里做重排序 |
| 处理层 | `pipeline.py` | 标准化、精确 URL 去重、时间过滤、基础清洗 |
| 缓存层 | `service.py` | 只缓存“已经过 Phase 2 基础清洗”的结果 |
| 排名层 | 新增 `ranking.py` 或放入 `service.py` | 负责 `Top News` 候选筛选、排序、Trending 输入准备 |
| 接口层 | `routes.py` | 输出 `items`、`top_news`、`trending`，并控制调试字段是否暴露 |

设计原则只有两条：
- `pipeline.py` 负责“把脏数据变干净”
- `ranking.py` 负责“从干净数据里挑重要结果”

不要把去重、排序、Trending 聚类混在同一层。

补充原则：
- `News` 视图只处理新闻源
- `SEC` 视图只处理 `sec_filing_rss`
- 不要再把披露流混进 `All`、`Top News`、`Trending`

## 三、Phase 2 要做的功能
### 3.1 精确去重前置
这是 Phase 2 的第一优先级。

目标：
- 不再依赖“读缓存时顺手修复”
- 改为 `force_refresh` 时先 exact dedup，再写缓存

实现方式：
1. 在 `pipeline.run()` 中启用 `deduplicate_exact_items()`
2. 插入位置固定为：

```text
normalize -> exact dedup -> age filter -> sort
```

规则：
- 只折叠 `canonical_url + source_type` 完全相同的记录
- 不合并相似标题
- 不合并同一事件的不同媒体报道
- 不合并不同 FilingId 的 SEC 条目
- 对于被多条 query 同时命中的同一篇新闻，不是简单“删除其余记录”，而是合并成一条主记录，并保留命中来源信息
- `sec_filing_rss` 已从新闻主流剥离，不参与新闻去重后的“主记录保留”比较

例子：
- `"https://reuters.com/a"` 被 `official_name` 和 `stock_news` 同时召回  
  -> 只保留一条
- Reuters 报道和 Yahoo Finance 报道标题很像，但 URL 不同  
  -> 仍保留两条
- 同一条 Quiver Quantitative 新闻同时被 `official_name`、`industry_news`、`stock_news`、`supporting_query` 命中  
  -> 前端只显示 1 条，后端保留它命中了 4 条 query 的信息
- 同一条正式新闻同时被 `company_news_rss` 和 Brave 命中  
  -> 优先保留 `company_news_rss` 作为新闻主记录，同时保留 Brave 的命中信息

这一步完成后：
- 缓存里保存的是“已去除 exact duplicate 的结果”
- Phase 1 为调试保留的重复命中展示，退出正式新闻页

推荐保留的合并字段：

```json
{
  "url": "https://...",
  "source_type": "brave",
  "matched_intents": ["official_name", "industry_news", "stock_news", "supporting_query"],
  "match_count": 4
}
```

这样可以同时满足两件事：
- 页面上不再被重复新闻污染
- 系统内部仍知道这条新闻是被哪些 query 命中的

如果前端最终只想显示一个 query 标签，不应在数据层丢掉其余命中信息。  
推荐做法是：
- 数据层保留全部 `matched_intents`
- 展示层额外派生一个 `primary_intent`

推荐展示优先级：

```text
official_name > stock_news > industry_news > supporting_query
```

也就是说：
- 数据层：全部保留
- UI 层：必要时只选一个主标签显示

### 3.2 Top News 正式化
Phase 1 的 `Top News` 只是“按时间取前 7 条”。  
Phase 2 开始给它正式规则，但不要一开始就做得很重。

建议拆成两步：

#### 第一步：候选池
从 `items` 中选出候选集，规则建议：
- 先要求有明确时间
- 排除明显不是新闻正文的残余项
- 限制单一来源或单一公司在候选池里的占比
- 只从新闻源中选，不从 `sec_filing_rss` 中选

#### 第二步：排序
候选池内再按分数排序。建议先只用这 4 个信号：
1. 发布时间新鲜度
2. 来源质量
3. query intent 权重
4. 是否命中多个来源

可以先定义一个很轻的分数：

```text
top_score =
  freshness_score
  + source_quality_score
  + intent_weight
  + source_count_bonus
```

例子：
- Reuters，2 小时前，来自 `official_name`  
  -> 高分
- 小站转载，5 天前，来自 `supporting_query`  
  -> 低分
- `sec_filing_rss` 的 `8-K` / transcript / filing entry  
  -> 不进入 `Top News` 候选池

#### 3.2.1 Top News Summary
Phase 2 的 `Top News` 应新增逐条 `summary`，但这个字段不能只是重复标题。  
它的目标是：在标题已经说明“发生了什么”的前提下，再补一句更短的“为什么值得看”。

推荐架构：

```text
top_news (7 items)
  -> summary input builder
  -> LLM summarizer
  -> summary cache
  -> top_news with per-item summary
```

规则：
- `Top News` 最终有 7 条，就生成 7 条逐条 summary
- 每条 summary 建议 1 句，最长不超过 40 到 60 个英文词
- 不允许只是把标题换个说法复述一次
- 应优先补充：
  - 财务结果/业务影响
  - 需求变化
  - 客户或行业背景
  - 为什么这条新闻会进入 `Top News`

推荐输入优先级：

**第一档：无正文版本（Phase 2 第一版先落地）**
- 输入只使用：
  - 标题
  - description/snippet
  - 来源
  - 发布时间
- 目标：
  - 不依赖正文抓取
  - 先保证每条 `Top News` 都能生成一条可读 summary

**第二档：正文增强版本（后续可选）**
- 如果后续拿到了正文或正文片段，再作为增强输入补给 LLM
- 这不是 Phase 2 第一版的前置条件

原因：
- 当前系统并不保证每条新闻都能拿到全文
- 如果强依赖全文，`Top News summary` 会和正文抓取绑死，落地成本过高

推荐缓存方式：
- 按文章稳定标识缓存 summary，优先使用：
  - `canonical_url`
  - 如果没有，再退回 `url`
- 只有当文章 URL 变了、或明确要求重新生成时，才再次调用 LLM
- 这样同一篇新闻在后续刷新中不会反复生成 summary

推荐缓存结构：

```json
{
  "article_key": "https://www.reuters.com/...",
  "summary": "...",
  "summary_source": "llm_from_metadata",
  "updated_at": "2026-04-17T10:30:00Z"
}
```

例子：
- 输入：
  - 标题：`Flex reports quarterly results`
  - description：`Revenue rose on stronger data center demand...`
- 不合格 summary：
  - `Flex reported quarterly results.`
- 合格 summary：
  - `Results were supported by stronger data center demand, making the update more relevant than a routine earnings headline.`

如果 LLM 失败：
- 不阻塞 `top_news`
- `summary` 返回空字符串或直接省略
- 前端可以短暂退回到 description，但 Final Version 仍应以生成式 summary 为主

#### 3.2.2 Top News Weekly 右侧整体总结
Phase 2 可以在前端 `Top News Weekly` 区块右侧增加一个整体总结模块，但它应总结整组 `Top News`，而不是挪用 `Trending`。

推荐内容：
- 1 段短总结，概括本轮 `Top News` 的主线
- 长度建议 2 到 3 句
- 重点回答：
  - 这一周/这一轮 `Top News` 在讲什么
  - 是财报、扩产、客户需求，还是供应链变化
  - 哪些公司或主题最突出

推荐数据流：

```text
top_news (7 items with summary)
  -> group summary builder
  -> LLM summarizer
  -> weekly_summary cache
```

例子：
- `Top News` 里如果同时包含：
  - FLEX 财报
  - JBL 扩产
  - 数据中心需求相关新闻
- 右侧整体总结可以写成：
  - `This week's Top News is dominated by earnings updates and expansion signals, with data center demand appearing as a recurring driver across multiple EMS names.`

这个模块建议独立命名，例如：
- `Top News Weekly Summary`
- `This Week in Top News`

不建议在这里直接放 `Trending` 内容。  
`Trending` 更适合作为下方独立话题区块。

范围规则：
- 当用户处于 `All` 视图时：
  - `Top News` 使用全局 `top_news`
  - 右侧整体总结也总结全局 `Top News`
- 当用户切到单一公司时：
  - `Top News` 改为该公司的 `top_news`
  - 右侧整体总结也只总结该公司的 7 条 `Top News`

也就是说，`Top News` 的逐条 summary 和右侧整体总结都必须跟随 `Top News` 的作用范围一起变化，不能继续停留在全局结果上。

### 3.3 Trending 变成话题列表
Phase 1 的 Trending 只是占位。  
Phase 2 的正式定义应是：

```text
Trending = 若干“话题簇”，不是文章列表
```

建议实现顺序：
1. 先做标题级聚类
2. 再决定是否需要 LLM 生成话题标题

技术路径建议：
- 第一版先用规则/相似度聚类
- 如果效果不够，再加 LLM 摘要

一个最小可用方案：
1. 取最近 N 天的新闻标题
2. 用标题相似度聚类
3. 每个 cluster 选一条代表文章
4. 输出：
   - `cluster_title`
   - `cluster_size`
   - `representative_item`

例子：
- `"Flex reports Q2 results"`
- `"Flex quarterly earnings beat estimates"`
- `"Flex posts better-than-expected Q2 revenue"`

这 3 条可以被归成一个 `Flex earnings` 话题簇。

输入约束：
- `Trending` 只使用新闻流输入
- 不消费 `sec_filing_rss`
- 年报、季报、earnings release 如果具有新闻价值，应由 Brave 或公司官网新闻流进入 `Trending`，而不是由 SEC 原始披露直接进入

#### 3.3.1 Trending 里应该放什么
Phase 2 的 `Trending` 不建议继续放“新闻条目 Top 10”，而应放“话题卡片”。

推荐每个话题卡片至少包含：
- `trend_title`
- `summary`
- `cluster_size`
- `representative_items`（1 到 3 条代表新闻）
- `companies`
- `updated_at`

也就是说，Trending 展示的重点不再是“哪篇文章排第几”，而是：
- 现在最热的话题是什么
- 这个话题影响到哪些公司
- 有多少新闻在讨论它

为了让前端含义更清楚，Phase 2 可以把 `Trending` 的第一版固定成与顶部主题筛选一致的 6 个主题卡：
- `Data Center`
- `Power`
- `Cooling`
- `Networking`
- `Semiconductor`
- `AI Infrastructure`

也就是说，第一版 `Trending` 可以理解成：

```text
6 个固定主题卡
  + 每张卡显示当前窗口内该主题命中的新闻数量
  + 每张卡显示代表新闻
```

这样用户更容易理解 `Trending` 在看什么，也能和顶部筛选主题形成一致心智。

补充约束：
- 第一版固定输出 6 张卡，不再只取命中数量最高的前 5 个主题
- 卡片顺序与顶部 preset 保持一致
- 每张卡都可以显示该主题当前使用的关键词，帮助用户理解“为什么这批新闻会被归到这里”
- 同一主题内部关键词为 `OR` 关系，不要求一条新闻命中全部关键词

例子：

```json
{
  "trend_title": "EMS earnings season",
  "summary": "Multiple EMS companies reported quarterly results within a short window, with coverage focusing on margin resilience and AI-linked demand.",
  "cluster_size": 9,
  "companies": ["FLEX", "JBL", "CLS"],
  "representative_items": [...]
}
```

前端展示建议：
- 主标题显示 `trend_title`
- 副标题显示 1 句 summary
- 旁边显示：
  - `3 companies`
  - `9 articles`
- 点击后展开代表新闻
- 第一版建议直接展示 6 张主题卡，而不是最多 5 张
- 每张卡都应有明确编号或明确的主题标题
- 数字必须带单位，例如：
  - `3 articles`
  - `2 companies`

补充约束：
- `Trending` 不应塞进 `Top News Weekly` 右侧总结模块
- `Trending` 的职责是展示“当前最热的话题簇”
- 右侧总结模块的职责是概括“当前 Top News 的共同主线”
- `Trending` 默认只在 `All` 视图展示
- 当用户切到单一公司时，建议隐藏 `Trending`，或显示“Trend 仅在 All 视图展示”的提示，而不是强行做一版单公司 `Trending`

### 3.4 公司专题增强 query
Phase 2 可以增加少量专题增强 query，但必须受控。

原则：
- 不替代现有 4 条基础模板
- 只做补充
- 每家公司最多加 1 到 3 条

建议增加字段：

```text
company_queries.query_tier = base | enhanced
```

这样：
- `base` = Phase 1 的 4 条正式模板
- `enhanced` = Phase 2 的专题模板

例子：
- `CLS`: `Celestica data center networking and storage news`
- `JBL`: `Jabil factory automation and robotics news`

### 3.5 SEC 从新闻主流剥离
Phase 2 应明确把 `sec_filing_rss` 从新闻主流里剥离出去。

规则：
- 点击 `All` 时，绝不显示 `sec_filing_rss`
- `Top News` 不包含 `sec_filing_rss`
- `Trending` 不包含 `sec_filing_rss`
- 各公司默认新闻列表也不显示 `sec_filing_rss`
- `sec_filing_rss` 不参与新闻源之间“保留谁作为主记录”的比较

原因：
- `sec_filing_rss` 本质上是披露流，不是新闻流
- `8-K`、transcript、附件、表格类文件会明显污染新闻阅读体验
- 如果年报、季报、earnings release 真的有新闻价值，Brave 或公司官网 RSS 通常也会覆盖

推荐产品形态：
- 默认视图：`News`
- 独立视图：`SEC Filings`

也就是说：
- `News` 负责新闻
- `SEC Filings` 负责披露文件

推荐交互：
- 在 `News Flash` 上方增加一个切换按钮或 tab：
  - `News`
  - `SEC Filings`
- 第一版建议用 tab / toggle，不建议先做复杂翻转动画

`SEC Filings` 视图规则：
- 只展示 `sec_filing_rss`
- 支持按公司筛选
- 不显示 `Top News`
- 不显示 `Trending`
- 不显示 `Data Center / AI / CapEx / Cooling` 这类新闻关键词筛选
- 只保留公司切换

这样可以把“新闻总览”和“披露文件”彻底分开，减少主新闻流噪音。

### 3.6 调试字段退出正式页面
Phase 1 为了观察召回路径，前端临时显示过：
- `intent`
- 重复命中

Phase 2 要明确收口：
- 默认新闻页不再显示重复命中调试信息
- `intent` 不再作为默认 UI 字段
- 如果仍需观察召回路径，只放到：
  - diagnostics
  - admin/debug view

### 3.7 排序逻辑分层
Phase 2 需要把“排序”拆成三层，不再混成一个大 sort：

1. **列表排序**
   - 用于 `items`
   - 规则：仍以时间倒序为主

2. **Top News 排序**
   - 用于重点新闻
   - 规则：时间 + 来源质量 + intent + source_count

3. **Trending 排序**
   - 用于话题簇
   - 规则：cluster_size + 新鲜度 + 覆盖公司数

推荐不要让一个统一分数同时决定这三类结果。  
因为：
- `items` 更适合时间序
- `top_news` 更适合“重要性序”
- `trending` 更适合“话题热度序”

例子：
- 一篇 Reuters 快讯可能在 `items` 中排很前
- 但如果只有单源且后续没有跟进，它未必应该排进 `Trending` 前列

## 四、返回结构建议
Phase 2 的返回结构仍然保持 Phase 1 风格，不要大换血。

建议：

```json
{
  "items": [...],
  "top_news": [...],
  "trending": [...],
  "sec_items": [...],
  "refresh_required": false,
  "diagnostics": {...}
}
```

字段规则：
- `items`：清洗后的平铺新闻列表，不含 `sec_filing_rss`
- `top_news`：正式候选和排序后的重点新闻
- `trending`：话题簇列表，不含 `sec_filing_rss`
- `sec_items`：独立 SEC 视图使用的披露列表
- `diagnostics`：默认可保留，但前端默认不必展开

## 五、前端升级点
### 5.1 Top News 卡片
Phase 2 的 Top News 卡片建议升级为：
- 标题
- summary
- 来源
- 发布时间
- 公司标签
- 可选图片

如果没有 summary：
- 回退到 description

展示要求：
- `summary` 放在标题下方，用灰色小字显示
- 每条 `Top News` 对应一条单独 summary
- `summary` 不能只是标题改写
- 整个 `Top News Weekly` 区块右侧增加整体总结模块
- 单公司视图下，`Top News` 卡片与其 summary 必须切到该公司范围
- `All` 视图下，再显示全局 `Top News`
- `Weekly Summary` 可以实现为折叠模块
- 折叠后不应继续占用右侧布局空间
- 标题建议直接使用 `Weekly Summary`

### 5.2 Trending 卡片
Phase 2 的 Trending 卡片建议从“文章排行”改为“话题卡片”：
- 话题标题
- 1 句 summary
- 公司数
- 文章数
- 代表新闻入口
- 第一版固定为 6 张卡，分别对应顶部 6 个主题 preset
- 卡片顺序固定，不再做“前 5 热门簇”这种不透明裁剪

范围建议：
- `All` 视图展示 `Trending`
- 单公司视图默认不展示 `Trending`
- 如果产品上不想完全隐藏，也应明确标注这是全局 `Trending`，不要让用户误以为它已经切到单公司范围
- `Trending` 也可以实现为折叠模块
- 折叠时不渲染卡片主体
- 卡片内的数字应明确带单位，例如 `3 articles`、`2 companies`
- 每张卡可以直接显示本主题的关键词说明

### 5.3 News / SEC 视图切换
Phase 2 前端建议把当前单一新闻面板拆成两个视图：
- `News`
- `SEC Filings`

默认进入 `News`。

`News` 视图：
- 展示 `Top News`
- 展示 `Trending`
- 展示 `News Flash`
- 支持关键词 preset

`SEC Filings` 视图：
- 只展示 `sec_items`
- 只保留公司筛选
- 不显示关键词 preset
- 不显示 `Top News`
- 不显示 `Trending`

推荐把切换入口放在 `News Flash` 上方或新闻主区顶部。

### 5.4 左上角关键词 preset
Phase 1 的前端 preset 只是轻量入口，Phase 2 第一版不要马上做成复杂关键词系统。  
推荐先做最短路径版本：

- preset 按钮不再只是“一个展示词”，而是一个可解释的小主题
- 每个主题允许带少量强相关隐藏关键词
- 隐藏关键词之间默认是 **OR** 关系，不是 **AND**
- 不做大词表维护
- 不调用 LLM 为 preset 自动生成扩展关键词
- 不对用户输入的关键词做自动扩写

这样做的目的不是最终效果最好，而是：
- 先让 preset 逻辑可解释
- 先看命中质量
- 避免一开始就因为扩展词过多，把结果弄脏
- 避免像 `Data Center` 这样因为扩展词太多，导致筛选逻辑难以理解和调试

关于主题选择，Phase 2 更适合从 Flex 及同类 EMS / infrastructure manufacturing 公司的业务语境出发，而不是继续保留过宽的通用词。

推荐第一版先用这 6 个主题：

1. `Data Center`
   - 隐性关键词：`data center`, `server`, `rack`, `storage`

2. `Power`
   - 隐性关键词：`power`, `power supply`, `power products`, `power systems`

3. `Cooling`
   - 隐性关键词：`cooling`, `liquid cooling`, `thermal`, `thermal management`

4. `Networking`
   - 隐性关键词：`networking`, `switch`, `router`, `interconnect`

5. `Semiconductor`
   - 隐性关键词：`semiconductor`, `chip`, `chips`, `silicon`

6. `AI Infrastructure`
   - 隐性关键词：`artificial intelligence`, `ai server`, `gpu`, `accelerator`, `inference`, `ai factory`

这 6 个主题更接近：
- 数据中心相关业务
- 电源 / 电力产品
- 散热与热管理
- 网络设备
- 芯片 / 半导体上下游

其中：
- 不建议保留过宽的 `AI`
  - 它太宽
  - 即使做词边界匹配，业务语义仍然太散
  - 如果确实需要，就改成更窄的 `AI Infrastructure`
- `CapEx` 也不建议作为第一版 preset
  - 它更像财务分析词，不像用户最直观的业务主题
- `Earnings` 暂时也不建议进入第一版 preset
  - 它更多是事件类型，不是核心业务方向

如果后续验证效果稳定，再进入第二步：
- 才考虑给每个 preset 增加少量扩展词
- 或把 preset 提到独立配置文件

也就是说，Phase 2 第一版的关键词筛选原则是：

```text
少量主题 + 少量强相关词 + OR 命中
```

先让用户知道自己筛的到底是什么，再决定是否进入第二轮关键词扩展。

补充规则：
- 选中某个 preset 后，命中该 preset 的任一隐藏关键词即可保留该新闻
- 不要求一条新闻同时命中该 preset 下的全部关键词
- 如果未来发现某个主题噪音太高，再缩减它的隐藏关键词，而不是继续扩写
- 对于 `AI Infrastructure` 这类主题，不建议只用裸词 `ai`
- 应优先匹配更明确的词或短语，例如：
  - `artificial intelligence`
  - `ai server`
  - `gpu`
  - `accelerator`
  - `inference`
- 前端应显示当前激活 preset 的实际关键词
- 推荐用一行浅色辅助文字展示，不直接塞进输入框

### 5.5 搜索输入框
当前筛选器里的输入框视觉上偏大。  
Phase 2 前端建议把它收窄成一个更短的辅助搜索框，不要继续占满整行。

UI 方向：
- 宽度固定或设置较小上限
- 仍保留手动输入能力
- 视觉权重低于 preset 按钮

目标：
- 让 preset 成为主入口
- 手动输入作为补充，而不是压过整排筛选器

补充规则：
- 手动输入框第一版不做 AI 自动扩词
- 用户输入什么词，就按什么词匹配
- 不要把输入框变成“输入一个词，系统自动替你扩成很多词”的黑盒行为
- 点击 preset 按钮时，不应再把一整串隐藏关键词直接写进输入框
- preset 的激活状态应独立于手动输入框
- 输入框只显示用户自己输入的词，不负责展示 preset 内部规则
- 搜索确认按钮只应在存在待应用输入时高亮
- 如果没有新的输入变化，按钮应弱化或禁用

## 六、实施顺序
Phase 2 不建议把功能平铺推进。  
更安全的做法是拆成 3 个必须串行完成的大部分：

### Part 1. 先稳定输入层
这是 Phase 2 的基础层。  
只有这部分完成后，后面的 `Top News` 和 `Trending` 才有稳定输入。

范围：
1. 精确去重前置到写缓存前
2. `pipeline.py` 保持只做清洗，不混入排序策略
3. `service.py` 写入的缓存，改为保存“已清洗、已 exact dedup”的结果
4. 把 `sec_filing_rss` 从默认新闻流中剥离
5. Phase 1 调试性重复展示退出正式页面

对应文件：
- `backend/news/pipeline.py`
- `backend/news/service.py`
- `backend/news/routes.py`
- `frontend/src/features/news/NewsPageFeature.tsx`

为什么必须先做：
- 不先去重，后面的 `Top News` 候选池会被重复项污染
- 不先稳定缓存，后面的 summary cache 和 trending cluster 都会建立在不稳定输入上

完成标志：
- `force_refresh` 写缓存前已 exact dedup
- 缓存中不再保留严格重复 URL
- `items` 默认不再混入 `sec_filing_rss`
- 前端不再靠 Phase 1 的重复项观察召回路径

### Part 2. 再做 Top News 正式层
这部分只处理“重点新闻”，不要和 `Trending` 混做。  
目标是先把最重要、最稳定、最容易感知价值的一块做好。

范围：
1. `Top News` 候选池
2. `Top News` 排序分数
3. 7 条逐条 LLM summary
4. summary cache
5. `Top News Weekly` 右侧整体总结模块

对应文件：
- 新增 `backend/news/ranking.py`，或先放入 `backend/news/service.py`
- 可新增 `backend/news/summary_cache.py` 或类似模块
- `backend/news/routes.py`
- `frontend/src/features/news/NewsPageFeature.tsx`

为什么排第二：
- `Top News summary` 应建立在稳定的 7 条结果之上
- 右侧整体总结应建立在已经稳定的 `Top News` 上
- 如果先做 `Trending`，很容易把资源分散到另一套尚未稳定的话题逻辑

完成标志：
- `top_news` 不再只是 `items[:7]`
- 7 条 `Top News` 都可以返回逐条 summary
- 同一文章的 summary 可以复用缓存
- 前端已能展示右侧整体总结模块
- `Top News` 已明确只从新闻流中选取

### Part 3. 最后做话题层和前端增强层
当前两部分完成后，再进入更容易扩散的“话题”和“筛选体验”。

范围：
1. `Trending` 话题聚类
2. `Trending` 话题标题与 summary
3. `News / SEC Filings` 视图切换
4. 前端关键词 preset 第一轮简化
5. 搜索输入框收窄
6. 专题增强 query
7. 重要性信号补强

对应文件：
- 新增 `backend/news/ranking.py` 或 `backend/news/trending.py`
- `backend/news/query_engine.py`
- `backend/news/routes.py`
- `frontend/src/features/news/NewsPageFeature.tsx`

为什么排最后：
- `Trending` 依赖更高层的聚类和话题表达，不适合作为 Phase 2 起点
- preset 和输入框属于前端交互增强，应建立在后端结果结构已稳定之后
- 专题增强 query 会改变召回分布，应该最后再加

完成标志：
- `Trending` 已是话题列表，不是文章列表
- 前端已支持单独 `SEC Filings` 视图
- preset 已从“扩展关键词”收紧到更可解释的第一版逻辑
- 输入框尺寸已收窄
- query 增强不再影响前两部分的基线判断

推荐的执行顺序就是：

```text
Part 1 稳定输入层
  -> Part 2 Top News 正式层
  -> Part 3 话题层与前端增强层
```

原因：
- 不先去重，`Top News` 和 `Trending` 的输入不可靠
- 不先稳定 `Top News`，后面很难判断哪些信号真的有用
- `Top News summary` 应建立在相对稳定的候选结果之上
- 右侧整体总结应建立在 7 条已稳定的 `Top News` 之上
- `Trending` 的话题结构确定后，前端卡片形态才能稳定
- 专题增强 query 应该建立在稳定的基础模板和排名逻辑之上

---
## 七、Phase 2 不做的事
为了避免 Phase 2 再次膨胀，先明确不做：
- 一次性做完事件级归并 + 语义聚类 + LLM 总结
- 把所有筛选逻辑配置化
- 自动发现新公司 RSS
- 推翻现有 4 intent 基础模板
- 把调试字段继续留在 Final Version 默认 UI
- 把 `sec_filing_rss` 再混回默认新闻主列表

---
## 八、完成标准
Phase 2 完成时，至少要满足：
1. `force_refresh` 写缓存前已经做 exact dedup
2. `items`、`Top News`、`Trending` 默认都不再混入 `sec_filing_rss`
3. 前端已提供独立 `SEC Filings` 视图，并支持按公司筛选
4. `Top News` 不再只是“按时间前 7 条”
5. `Top News` 的 7 条新闻都可生成逐条 LLM summary，且同一文章支持缓存复用
6. `Top News Weekly` 右侧已有整体总结模块
7. `Trending` 已经是话题列表，不是文章列表
8. 前端 preset 已完成第一轮简化，先采用精确词匹配
9. 筛选器输入框已收窄，不再压过 preset 主入口
10. Phase 1 调试字段退出正式新闻页面
11. 新增公司仍可沿用统一模板规则，不需要重新定义 Phase 1 基线

如果这 11 条没有做到，就说明 Phase 2 还没有真正完成。
