## 一、Phase 4 的定位
`Phase 4` 的作用不是回头修改 `Phase 3` 的必做项，而是承接后续这些类型的需求：

1. 更偏体验增强、不是当前交付闭环必需的前端优化
2. 更复杂的排序、拖拽、布局与视觉增强
3. 更高阶的历史分析、趋势分析或聚类增强
4. 在 `Phase 3` 基线稳定后，再追加的辅助功能

标准口径：

```text
Phase 3 先完成当前可交付版本；
Phase 4 再承接后续增强项。
```

---
## 二、与 Phase 3 的边界
当前建议边界如下：

### 仍留在 Phase 3 的内容
- 新增公司流程闭环
- 删除公司与缓存清理
- 单公司初始化抓取
- 官网确认
- `Summary` 升级
- `Trending` 升级
- `cross-source merge`
- 当前 Part 1 ~ Part 4 的主线能力

### 更适合后续进入 Phase 4 的内容
- 顶部公司 chip 的拖拽排序
- 更复杂的 chip 动画与交互细节
- 更高级的 icon / favicon 展示优化
- 更严格的历史新闻仓库与长窗口趋势统计
- 更复杂的趋势聚类与主题演化分析
- 其他不影响 `Phase 3` 当前交付闭环的增强能力

## 四、后续补充原则
后续如果再出现新需求，可按下面规则判断放到哪里：

1. 如果它影响 `Phase 3` 当前交付闭环
   - 继续放在 `Phase 3`

2. 如果它主要是增强体验、复杂排序、额外分析能力
   - 优先放到 `Phase 4`

3. 如果它会明显扩大当前实现范围，容易拖慢 `Phase 3`
   - 默认放到 `Phase 4`
