## 一、Phase 3 的目标
Phase 3 主要处理 4 类升级：

1. 完成更接近最终产品形态的顶部入口与新增公司流程
2. 把 `Top News` / `Weekly Summary` 升级成更稳定的 LLM 总结层
3. 把 `Trending` 从 Phase 2 的固定主题卡升级成真正的趋势总结
4. 在不推翻 Phase 2 的前提下，补上跨来源重复合并与前端布局收口

这一阶段的特点是：
- 改动更多落在前端体验与交互
- 会引入更明确的 LLM 使用边界
- 会新增少量后端辅助模块，但不重做新闻主链路
- 所有升级都建立在 `Phase 2` 的输入清洗、缓存和 SEC 剥离基线之上

---
## 二、总体原则
Phase 3 建议遵守这 5 条原则：

1. **不推翻 Phase 2**
   - 不重写 `Phase 2` 的 `Top News` 范围规则
   - 不把 `SEC` 混回新闻主流
   - 不改变 `Trend` 只在 `All` 视图展示的 Phase 2 基线，除非后续单独升级

2. **前端入口增强优先于底层重构**
   - 先把用户真正会使用的入口做完整
   - 再做趋势总结和高级合并

3. **LLM 只做识别与总结**
   - 允许用于公司识别、summary、trend title / summary
   - 不用于新闻抓取、strict dedup、官网直接落库、logo 地址猜测

4. **高级合并单独成层**
   - `exact dedup` 仍然是 Phase 2 的基础层
   - `cross-source merge / event-level merge` 在 Phase 3 单独设计

5. **每个 Part 必须可验收**
   - 必须先完成 `Part 1`
   - 再进入 `Part 2`
   - 再进入 `Part 3`
   - 最后才做 `Part 4`

## 三、建议架构
Phase 3 不需要重做新闻底层链路，建议在现有架构上加 3 层薄增强：

```text
frontend entry
  -> resolver / suggest
  -> existing news pipeline
  -> summary / trend upgrade
  -> frontend presentation
```

职责拆分建议：

| 层 | 建议文件 | Phase 3 职责 |
|---|---|---|
| 顶部入口层 | `frontend/src/features/news/NewsPageFeature.tsx` 或拆分子组件 | 现有公司 chip + 新增公司输入框 + autocomplete 候选 |
| 识别层 | 可新增 `backend/news/company_resolver.py` | 标准公司候选识别、官网候选补全、ticker 标准化 |
| 现有链路层 | 继续复用 `onboarding.py` / `service.py` / `query_engine.py` | 生成模板、保存公司、抓取新闻、写缓存 |
| 总结升级层 | 可新增 `backend/news/summary_service.py` / `backend/news/trend_summary_service.py` | 单条 summary、weekly summary、trend summary |
| 高级合并层 | 可新增 `backend/news/merge_service.py` | cross-source merge / canonical_event_key |
| 展示层 | 前端新闻页组件 | summary、trending、折叠、右侧面板、关键词显示 |

## 五、顶部公司入口与新增公司流程

### 5.1 顶部现有公司入口保留不重做
当前顶部这一排公司缩写 chip 已经可用，且视觉效果满足需求。  
Phase 3 不要求重做它，只要求：

- 保留当前这排现有公司缩写组件
- 当前正式展示对象是 **6 家公司**，不是 36 家
- 只在这一排最右侧追加新增公司输入框

也就是说，这里的修正是：

```text
"36 家公司"
  -> 改为
"当前顶部展示 6 家现有公司 + 右侧新增输入框"
```

### 5.1.1 顶部公司入口的删除权限
Phase 3 需要把“哪些公司可以删除”写清楚。

第一版建议：
- 当前默认内置的 6 家公司
  - 视为系统基线公司
  - **不可删除**
- 用户后续通过新增公司输入框加入的公司
  - 视为用户扩展公司
  - **可删除**

标准口径：

```text
内置 6 家公司不可删除；
只有用户后续新增的公司才显示删除入口。
```

这条规则应同时体现在：
- 后端公司元数据
- 前端 chip 交互
- 删除接口的权限检查

推荐元数据字段：

```json
{
  "ticker": "FLEX",
  "is_builtin": true,
  "is_deletable": false
}
```

```json
{
  "ticker": "GOOG",
  "is_builtin": false,
  "is_deletable": true
}
```

### 5.2 新增公司输入框的位置与形态
新增公司输入框应放在现有公司 chip 这一排的最右侧。

建议形态：
- 与现有 chip 保持同一行
- 高度与现有 chip 基本一致
- 输入时在下方弹出候选列表
- 候选列表样式应尽量接近现有 chip / 小卡片语言

目标不是普通搜索，而是：

```text
输入一家新的上市公司
  -> 返回标准候选公司对象
  -> 用户确认
  -> 接入现有新闻链路
```

### 5.2.1 可删除公司的前端交互
删除功能不仅要有后端接口，也必须有前端入口。

第一版建议：
- 对不可删除的内置 6 家公司
  - 不显示删除图标
- 对可删除的新增公司
  - 在 chip 右上角显示一个小关闭图标
  - 或在 hover / focus 时显示该图标

推荐交互：
1. 默认状态下，新增公司 chip 与普通 chip 基本一致
2. 鼠标悬停或聚焦时，右上角出现一个小叉 / close icon
3. 点击该图标后，弹出删除确认
4. 用户确认后才执行删除

不建议：
- 要求用户“悬停几秒”才显示删除图标

更稳的口径应是：

```text
hover / focus 即可显示删除入口，
不要求等待数秒。
```

这样更符合常见前端交互，也更容易实现和测试。

### 5.2.2 顶部公司入口的默认排序
Phase 3 需要明确新增公司加入顶部入口后的排序规则，避免出现：
- 新增 `Google` 后插进 `FLEX` 和 `JBL` 中间
- 不同刷新后顺序漂移

第一版推荐规则：

1. 内置 6 家公司顺序固定
   - 保持当前既有顺序
2. 新增公司默认追加到内置公司之后
3. 多个新增公司之间，默认按添加时间顺序排列

标准口径：

```text
内置公司顺序固定；
新增公司默认追加到末尾；
不按字母顺序自动插入已有内置顺序中。
```

这样 `Google` 不会自动插到 `FLEX` 和 `JBL` 之间。

### 5.2.3 是否支持手动重排
Phase 3 可以支持手动调整顶部公司 chip 顺序，但要把优先级写清楚。

建议分两步：

#### 第一版必做
- 不要求拖拽排序
- 只要求：
  - 内置 6 家顺序固定
  - 新增公司追加到末尾
  - 删除后其余 chip 顺序稳定

#### 第二版可选增强
- 支持手动拖拽调整顺序
- 例如把 `Google` 拖到 `JBL` 后面

如果后续做拖拽，建议规则为：
- 内置 6 家公司之间仍不允许互相拖乱系统默认顺序
- 允许把新增公司插入到指定位置
- 前端拖拽后，应把顺序持久化到后端元数据中

也就是说，Phase 3 第一版最稳的要求应是：

```text
先解决默认排序稳定；
拖拽重排属于可选增强，不作为 Part 1 的硬依赖。
```

### 5.3 输入方式：必须做成 autocomplete / suggest
Phase 3 第一版不应是“输入后直接提交”的普通输入框，而应做成：

```text
输入即联想
```

推荐支持输入：
- 公司正式名
- ticker
- 常见英文简称
- 中文名

示例：
- `NVIDIA`
- `英伟达`
- `NVDA`
- `Google`
- `Alphabet`

推荐流程：
1. 用户在顶部新增输入框里输入内容
2. 前端在输入过程中实时请求候选结果
3. 下拉区域展示 3 到 8 条候选公司对象
4. 用户点击某一条候选结果
5. 再进入确认与保存流程

如果没有 autocomplete，用户输入 `NVIDIA` 时，系统可能把一个未标准化字符串直接当成新公司对象处理，这会带来误识别风险。

### 5.3.4 suggest 的实现策略
`suggest` 不能简单理解为“每次击键都调一次 LLM”。  
Phase 3 第一版应明确采用分层策略：

```text
本地轻量 suggest
  -> 不足时再降级到 LLM suggest
  -> 最终点击候选后再进入 resolve
```

推荐顺序：
1. 先查一小份内置“常用美股上市公司 alias 字典”
   - 目标不是覆盖全市场
   - 只要求先覆盖最常见、最容易被手动新增的公司
   - 例如：
     - `Google / Alphabet / 谷歌 -> GOOGL`
     - `NVIDIA / NVDA / 英伟达 -> NVDA`
     - `Intel / INTC / 英特尔 -> INTC`
     - `AMD / 超威 / 超微 -> AMD`
2. 再查本地已注册公司列表
   - 现有 ticker
   - 现有 display_name
   - aliases
3. 再做轻量模糊匹配
   - 前缀匹配
   - 子串匹配
   - 简单相似度匹配
4. 只有本地结果仍然不足时，才调用 LLM 生成补充候选
5. 用户点击候选后，再调用 `resolve`

也就是说，Phase 3 第一版不追求“完整证券主数据平台”，而是追求：

```text
先让常见美股上市公司可用
再让 LLM 去补长尾公司
```

这个“常用 alias 字典”可以很小，但应至少覆盖：
- Google / Alphabet
- NVIDIA
- Intel
- AMD
- Amazon
- Microsoft
- Apple
- Meta

如果后续需要更广覆盖，再接 SEC / 第三方 profile 数据源；Phase 3 第一版不强制引入更重的数据接入。

第一版建议规则：
- 输入长度 `< 2`
  - 不调用 LLM
  - 只做本地候选或不返回结果
- 本地已命中 `>= 3` 条高相关候选
  - 不调用 LLM
- 本地命中为空或明显不足
  - 才允许调用 LLM 生成补充候选

这样可以避免：
- 每次击键都打 LLM，成本过高
- autocomplete 延迟过大
- `suggest` 和 `resolve` 两层都过度依赖 LLM

标准口径应是：

```text
suggest = 常用 alias 字典 + 本地候选优先，LLM 只做补充
resolve = 用户选定候选后的标准化确认
```

### 5.3.1 输入预处理：先限缩，再交给 resolver
新增公司输入框不应把用户原始整句直接发给 LLM。  
Phase 3 第一版应先做一层轻量输入预处理，把输入限制为“公司线索”，再交给 resolver。

允许直接进入 resolver 的输入类型：
- ticker
- 公司正式名
- 常见英文简称
- 中文名

示例：
- `PLXS`
- `Plexus`
- `Plexus Corp`
- `英伟达`
- `NVIDIA`
- `NVDA`

不建议原样直传给 resolver 的输入类型：
- `plxs公司的官网是什么`
- `帮我查一下英伟达的官方网站`
- `我想添加一家叫 Google 的公司`

推荐前端预处理规则：
1. 去掉首尾空格
2. 折叠多余空白
3. 去掉明显问句和操作性前缀，例如：
   - `帮我查一下`
   - `官网是什么`
   - `我想添加`
   - `请帮我找`
4. 只保留最可能的公司线索短语
5. 将清洗后的短语传给 resolver

目标是把：

```text
用户自然语言输入
  -> 压缩成公司线索
  -> 再做标准化识别
```

例如：

```text
"plxs公司的官网是什么"
  -> "PLXS"

"我想添加一家叫 Google 的公司"
  -> "Google"
```

这样可以明显降低 LLM 在长句理解上的误差。

### 5.3.2 输入边界与异常情况
为了防止特殊输入把 resolver 变成开放式问答，Phase 3 应明确这些边界：

1. 输入长度限制
   - 推荐限制为 `2 ~ 64` 个字符
   - 超过上限时先提示用户精简输入，不直接发给 resolver

2. 字符类型限制
   - 允许中英文、数字、空格、常见公司缩写符号
   - 例如：`.` `-` `&`
   - 不建议允许长段标点、URL、邮箱、完整句子原样进入 resolver

3. 空输入 / 纯噪音输入
   - 如空字符串、只有标点、只有“官网”“添加公司”这类词
   - 不调用 resolver，直接提示重新输入

4. 多公司输入
   - 如 `NVIDIA AMD Intel`
   - 第一版不做多公司并行解析
   - 前端提示“请一次只输入一家公司”

5. 明显非公司对象
   - 如 `data center`
   - `AI`
   - `earnings`
   - `server`
   - 这类更像主题词而不是公司名
   - 不直接进入保存流程，应提示“请输入公司名或 ticker”

6. URL 输入
   - 如用户直接输入 `https://www.nvidia.com`
   - 第一版可允许进入 resolver 作为辅助线索
   - 但不能直接把域名当公司对象保存，仍需返回标准公司候选并确认

### 5.3.3 resolver 的责任边界
前端预处理和 resolver 的分工应明确：

- 前端负责：
  - 去噪
  - 限长
  - 阻止明显非法输入
  - 保证一次只送一家公司线索

- resolver 负责：
  - 公司标准化识别
  - ticker 推断
  - 官网候选补全
  - confidence / reason 输出

标准口径应是：

```text
前端先把输入限制在“公司线索”范围内，
resolver 再对该线索做结构化识别。
```

### 5.4 `company_resolver.py` 的职责
建议新增 `backend/news/company_resolver.py`，至少承担两件事：

1. `suggest`
   - 根据用户输入返回候选公司对象列表

2. `resolve`
   - 对用户最终选中的候选对象做标准化与官网补全

推荐最小返回结构：

```json
{
  "company_name": "NVIDIA Corporation",
  "display_name": "NVIDIA",
  "ticker": "NVDA",
  "aliases": ["NVIDIA", "英伟达"],
  "official_website": "https://www.nvidia.com",
  "official_domain": "nvidia.com",
  "confidence": 0.92,
  "reason": "NASDAQ-listed company; ticker and official site matched"
}
```

### 5.5 `company_resolver.py` 的 prompt 规范
`company_resolver.py` 如果使用 LLM，文档里必须写清楚 prompt 目标，避免程序员各写各的。

最低要求：
- 输入是一段公司线索
  - 公司名、ticker、简称或中文名
- 输出必须是标准公司对象，不是自由文本
- 必须优先输出：
  - `company_name`
  - `ticker`
  - `official_website`
  - `aliases`
  - `confidence`
  - `reason`
- 不能直接返回模糊描述
- 不能凭空生成与上市公司不一致的 ticker

建议 prompt 目标写死为：

```text
你是上市公司基础信息标准化助手。
根据输入线索识别最可能的上市公司，
返回标准公司对象，不要返回解释性散文。
如果无法高置信度确认，降低 confidence，并保留 reason。
```

### 5.5.1 `confidence` 阈值与前端确认规则
`confidence` 不能只作为 LLM 自报分数存在，Phase 3 必须给出可执行阈值。

建议第一版写死为：

- `confidence >= 0.85`
  - 允许进入正常确认流程
  - 前端展示候选公司卡，用户可直接点击确认保存
- `0.60 <= confidence < 0.85`
  - 视为中置信度
  - 前端必须要求人工确认后才能保存
  - 确认卡片中必须展示 `company_name / ticker / official_website / reason`
- `confidence < 0.60`
  - 视为低置信度
  - 不允许直接保存
  - 前端应提示用户重新输入、改用 ticker、或手动补充公司信息

第一版不为不同公司类型设置不同阈值，统一使用这一套标准，避免程序员各自实现不同逻辑。

示例：

```json
{
  "company_name": "Alphabet Inc.",
  "ticker": "GOOGL",
  "official_website": "https://abc.xyz",
  "confidence": 0.62,
  "reason": "Input 'Google' matched listed parent company Alphabet"
}
```

这类结果不能直接保存，前端必须进入人工确认流程。

### 5.6 官网地址：初始可空，但保存前应确认
新增公司时，官网地址在初始输入阶段可以为空，不要求用户手填。

推荐顺序：
1. 用户先输入公司名 / ticker / 简称
2. 系统返回标准公司候选对象
3. 官网地址由规则层或 LLM 补全
4. 用户确认官网后再正式保存

也就是说：
- **初始输入时官网可以为空**
- **保存前官网应尽量被补全并确认**

### 5.7 官网获取方式
官网获取不建议只依赖某一个 Python 库，也不建议全靠 LLM。

推荐顺序：
1. 先查已有公司映射或别名字典
2. 再用规则工具或搜索型工具推导官网候选
3. 最后由 LLM 做标准化补充
4. 用户确认最终官网

标准口径应是：

```text
官网候选可以由规则工具 + LLM 共同补全，
但最终官网值必须经过确认后再写入。
```

### 5.8 Favicon / 图标规则
如果后续需要把新增公司也显示成和现有入口一致的视觉单元，图标规则应明确：

- 不让 LLM 猜 logo 地址
- 从确认后的 `official_website` 解析 `official_domain`
- 按规则构造 favicon URL

推荐第一版规则：

```text
icon_url = https://www.google.com/s2/favicons?domain={official_domain}&sz=64
```

或统一使用你们选定的 favicon 提供方式。  
如果 favicon 拉取失败：
- 退回到 ticker 占位块
- 不阻止新增公司保存

### 5.9 接入现有新闻链路
新增公司不应单独发明一条新链路，而是：

```text
company_name / ticker / official_website
  -> preview / 正式模板生成
  -> 保存 company_queries
  -> 使用 Brave 抓新闻
  -> 后续进入缓存和展示
```

### 5.9.1 新增公司后的刷新：必须做单公司增量抓取
Phase 3 需要明确一条资源控制规则：

```text
新增公司成功后，
不允许要求用户再点击一次“全局刷新”去让所有公司重新抓新闻。
```

原因：
- 当前页面只有一个刷新按钮
- 如果新增 `Google` 之后只能靠全局刷新生效
- 那么系统会把现有公司全部重新请求一遍
- 这会额外消耗 Brave 配额、RSS 抓取次数，以及 SEC 访问额度

Phase 3 第一版应改成：

1. 新增公司保存成功后
2. 后端只对**该新公司**执行一次初始化抓取
3. 只写入该公司的缓存与相关 summary/trend 输入
4. 前端刷新公司列表与该公司视图
5. 不触发其他公司全量 refresh

推荐接口语义：

```text
POST /api/news/companies
  -> 创建公司配置
  -> 仅为该 ticker 执行一次 initial fetch
```

如果初始化抓取较慢，可允许异步化，但语义仍应是：

```text
新增公司 = 单公司初始化抓取
不是全局新闻重抓
```

前端的全局刷新按钮仍然保留，但它不应成为“新增公司后必须点一下”的依赖步骤。

### 5.9.2 SEC / RSS 访问控制
因为 `SEC` 源和部分 RSS 源对访问频率更敏感，Phase 3 第一版应同时写清楚：

- 单公司初始化抓取时，只抓该公司的：
  - Brave
  - `company_news_rss`
  - `sec_filing_rss`
- 不因为新增一家公司而顺带重抓其余公司 RSS / SEC
- 不因为 UI 上点击一次“刷新”就无差别重抓全部公司的 SEC feed

标准口径应是：

```text
新增公司流程优先走单公司增量抓取；
全量刷新是单独动作，不应被新增公司流程隐式触发。
```

### 5.9.3 删除公司功能：Phase 3 必须补齐
Phase 3 不能只有“新增公司”而没有“删除公司”。

因为如果只增不删，会产生 3 个问题：
- 顶部公司入口会不断累积
- 公司筛选列表会不断膨胀
- 该公司的缓存、summary cache、trend 输入数据会一直残留

所以 Phase 3 第一版应明确增加：

```text
删除公司 = 配置删除 + 缓存清理 + 摘要清理
```

### 5.9.4 删除公司时必须清理什么
第一版最小清理范围应包括：

1. 公司配置
   - company registry / company_queries / 相关元数据

2. 新闻缓存
   - `data/news_cache/company_{ticker}.json`

3. 与该公司直接绑定的 summary / weekly summary cache
   - 如果按 ticker 分区存储，直接清理该 ticker 下全部摘要
   - 如果按 article key 全局存储，至少清理与该公司 cache 建立的索引关系

4. 与该公司绑定的 trend / trend summary 相关数据
   - 该公司不再参与后续 trend 输入集合
   - 如存在 trend summary cache、trend cluster cache 或事件索引，也必须清理与该 ticker 绑定的记录

5. 该公司的 `sec_items` / `SEC Filings` 视图数据

6. 顶部入口与公司筛选中的展示项

标准口径应是：

```text
删除公司后，
该公司不应继续出现在顶部入口、筛选器、News Flash、SEC Filings、
以及与其绑定的 summary / weekly summary / trend 结果中。
```

### 5.9.4.1 删除公司的验收标准
Phase 3 对删除公司的验收不能只看前端“看不见了”，还必须验证后端与缓存层没有残留。

第一版建议至少做这 3 层检查：

1. 前端展示层
   - 顶部公司 chip 已消失
   - 公司筛选器中已无该 ticker
   - `News Flash` / `SEC Filings` 视图中不再出现该公司入口

2. 后端配置层
   - 公司 registry / company_queries / 相关元数据中已无该 ticker
   - 该公司不再参与后续刷新、聚合、summary、trend 输入

3. 缓存与持久化层
   - `data/news_cache/company_{ticker}.json` 已删除
   - 该公司的 summary / weekly summary 缓存已清理
   - 该公司的 trend / trend summary 相关缓存与索引已清理
   - 如存在与该公司绑定的索引或关联记录，也必须一并移除

推荐验收示例：

```text
新增 Google
  -> 生成 company_GOOG.json
  -> 生成该公司的 summary 缓存

删除 Google 后
  -> 前端不再显示 Google
  -> 后端 company registry 中无 GOOG
  -> company_GOOG.json 不存在
  -> 与 GOOG 绑定的 summary / weekly summary / trend 结果不再可读
```

### 5.9.4.2 删除后的隔离要求
Phase 3 还应明确一条隔离要求：

```text
被删除公司的旧缓存内容，
不应在后续刷新或聚合过程中“串”到其他公司结果里。
```

也就是说，删除 `Google` 后：
- 不只是“不再读取 Google 自己的缓存文件”
- 还要保证该公司的旧新闻不会继续出现在：
  - 其他公司的 `items`
  - `All` 聚合视图
  - `Top News`
  - `Trending`
  - `Weekly Summary`

第一版可执行要求：
1. 删除动作完成后，该 ticker 不再出现在聚合输入集合中
2. 聚合接口重新读取公司列表时，不再包含被删除 ticker
3. 删除后再次触发刷新、summary 生成或 trend 聚合，不应再从任何路径读到该公司的旧缓存

### 5.9.5 删除动作的安全策略
删除公司不应做成“点一下立即无提示删除”。

第一版建议：
1. 用户点击删除
2. 前端弹出确认框
3. 明确提示将删除：
   - 公司入口
   - 公司新闻缓存
   - 该公司的摘要缓存
4. 用户确认后再执行

如果后续担心误删，还可以要求输入 ticker 二次确认，但第一版不强制。

### 5.10 Brave `count=50` 的升级范围
`Phase 1` 默认 `count = 40`，`Phase 2` 没有继续改。  
进入 `Phase 3` 后，应把系统正式模板的 Brave 默认抓取上限统一提升到：

```text
count = 50
```

这条统一升级范围必须写清楚：
- 新增公司通过 resolver / onboarding 生成的默认模板
- 现有 registry / companies 中沿用正式模板规则的公司
- 后续新增公司入口落地后自动生成的标准模板

这里的意思是：
- 每条 Brave query 单次最多请求 50 条结果
- 不是页面最终只显示 50 条
- 也不是一家公司总共只抓 50 条

## 六、Summary 升级层

### 6.1 目标
Phase 3 的 summary 升级主要解决：
- 避免 summary 只是标题改写
- 让单条 summary 更像“为什么值得看”
- 让右侧整体总结更像 briefing
- 提高不同新闻之间的摘要风格一致性

### 6.2 单条 Top News Summary
继续沿用 `Top News` 的核心排序结构，但升级 prompt 和缓存表达。

同时明确作用范围规则：

- `All` 视图
  - `Top News` 继续从全局候选池中排序
  - 最多展示 `7` 条
- 单公司视图
  - `Top News` 只从最近 `7d` 的该公司候选新闻中选取
  - 不再强制凑满 `7` 条
  - 如果最近一周只有 `5` 条符合条件，就只展示 `5` 条

标准口径：

```text
All 视图 = 最多 7 条全局 Top News；
单公司视图 = 最近 7 天内的公司 Top News，不强行补足到 7 条。
```

推荐输入：
- 标题
- description
- 来源
- 发布时间
- 如后续拿到正文片段，可作为增强输入

prompt 目标：
- 不要重复标题
- 优先补充业务影响、行业背景、客户背景
- 控制为 1 句

### 6.3 Weekly Summary
右侧整体总结模块继续保留，但语言质量升级。

推荐输入：
- 当前作用范围内的 Top News 标题
- 当前作用范围内的单条 summary
- 公司分布
- 主题分布

输出目标：
- 2 到 3 句
- 像 weekly brief，不像标题拼接

补充说明：
- 如果单公司视图下只有 `5` 条 Top News
  - weekly summary 就基于这 `5` 条生成
- 不要求为了凑满 `7` 条而回填更早的旧新闻

### 6.3.1 Weekly Summary 的输入降级规则
Phase 3 必须明确 weekly summary 的二层依赖处理方式：

```text
文章
  -> 逐条 article summary
  -> weekly summary
```

如果某篇文章的逐条 summary 失败，weekly summary 不能直接吃空字符串。  
第一版必须采用可执行的降级规则：

1. 单条 summary 成功
   - weekly summary 使用该条 `summary`
2. 单条 summary 失败，但有 `description`
   - weekly summary 使用 `description` 作为 fallback 输入
3. 单条 summary 失败，且 `description` 也为空
   - weekly summary 使用 `title` 作为最低保底输入

也就是说，weekly summary 的输入优先级应固定为：

```text
summary > description > title
```

不允许把失败文章以空字符串传入 weekly summary prompt。

### 6.3.2 Weekly Summary 的输入完整度与失败规则
为了避免“看起来生成成功，但质量已无声下降”，Phase 3 应明确记录 weekly summary 的输入完整度。

推荐最小字段：

```json
{
  "weekly_summary": "...",
  "summary_version": "v2",
  "summary_status": "fallback_used",
  "summary_source": "llm_with_article_summaries",
  "source_item_count": 7,
  "summary_ready_count": 5,
  "fallback_count": 2
}
```

解释：
- `source_item_count`
  - 本次参与 `Top News` 的文章数
- `summary_ready_count`
  - 逐条 summary 成功生成的文章数
- `fallback_count`
  - 使用 `description` 或 `title` 作为 fallback 输入的文章数

第一版建议规则：

- `fallback_count = 0`
  - `summary_status = ready`
- `fallback_count > 0` 且 `summary_ready_count >= 4`
  - `summary_status = fallback_used`
- `summary_ready_count < 4`
  - 视为输入缺失过多
  - weekly summary 可直接标记为 `failed`
  - 前端回退为“不显示整体总结”或显示简短提示，不输出误导性总结文本

示例：

```text
7 条 Top News 中有 2 条 article summary 失败，
系统仍可用其 description 生成 weekly summary，
但状态必须写成 fallback_used，
而不是伪装成完全正常的 ready。
```

### 6.4 `summary_version` 初始值与迁移规则
Phase 3 不能只写“引入 `summary_version`”，还要写清楚初始值和迁移规则。

建议：
- Phase 2 已有摘要统一视为 `summary_version = v1`
- Phase 3 新生成摘要统一写为 `summary_version = v2`

迁移规则：
1. 读取缓存时，如果有摘要但没有 `summary_version`
   - 视为 `v1`
2. 如果页面要求使用 `v2`
   - 后端可按需重算
3. 不要求一次性重算全部旧摘要
   - 可采用 lazy migration

### 6.5 summary 状态字段
Phase 3 文档应明确 API 响应和前端展示状态。

建议最小状态：
- `ready`
- `failed`
- `fallback_used`

推荐结构：

```json
{
  "summary": "...",
  "summary_version": "v2",
  "summary_status": "ready",
  "summary_source": "llm_with_context"
}
```

weekly summary 也应采用同样的状态表达，并补充输入完整度字段：

```json
{
  "weekly_summary": "...",
  "summary_version": "v2",
  "summary_status": "fallback_used",
  "summary_source": "llm_with_article_summaries",
  "source_item_count": 7,
  "summary_ready_count": 5,
  "fallback_count": 2
}
```

### 6.6 Cache 继续复用
Phase 3 不推翻 Phase 2 的 summary cache，只在其上增强。

推荐 article summary cache 结构：

```json
{
  "article_key": "https://www.reuters.com/...",
  "summary": "...",
  "summary_version": "v2",
  "summary_status": "ready",
  "summary_source": "llm_with_context",
  "updated_at": "2026-04-17T10:30:00Z"
}
```

## 七、Trending 升级

### 7.1 Phase 3 对 Trending 的重新定义
Phase 2 的 `Trending` 更像固定主题卡。  
进入 Phase 3 后，`Trending` 应回答：

```text
最近一段时间里，新闻流里反复出现、值得被总结出来的主要趋势是什么？
```

而不是：

```text
哪些新闻命中了预设主题词？
```

### 7.2 输入范围
`Trending` 只使用新闻流：
- Brave
- `company_news_rss`

不使用：
- `sec_filing_rss`

### 7.3 与 Phase 2 age filter 的衔接
Phase 3 不重新定义全局时间过滤，而是在 Phase 2 输入基线上做趋势分析。

仍然沿用：
- `Brave`：最近 5 个月
- `RSS`：最近 6 个月

`7d / 30d / 60d` 的作用是：
- `7d`：看短期热度
- `30d`：看近一个月是否反复出现
- `60d`：看这是不是持续性方向

也就是说：
- Phase 2 age filter
  -> 决定哪些新闻能进入主新闻流
- Phase 3 trend windows
  -> 决定趋势分析如何观察短期与中期现象

### 7.3.1 当前 30d / 60d 的可实现边界
Phase 3 文档需要明确一条现实约束：

- 当前系统的公司新闻存储是“每家公司一份当前快照 JSON”
- 这份快照有 `48h` freshness 判断
- 但快照内部保留的新闻项本身，仍可能覆盖最近 5 个月 / 6 个月

因此，Phase 3 的 `30d / 60d` Trending **不是历史全量仓库意义上的严格趋势统计**，而是：

```text
基于当前快照中仍然存在的新闻项，
对 30d / 60d 做近似趋势分析。
```

这意味着：
- 可以按当前快照里的 `published` 时间去统计 `7d / 30d / 60d`
- 但不能保证这代表过去 30 / 60 天内“所有曾经出现过的新闻”
- 某些 20 天前或 45 天前的新闻，如果这次刷新没有重新回到快照里，就不会参与趋势统计

Phase 3 第一版可以接受这种近似实现，因为它仍能支持：
- 趋势标题生成
- 趋势 summary
- 近一段时间反复出现现象的粗粒度判断

但如果后续要做更严格的 `30d / 60d` 趋势统计，就需要新增历史存储层，例如：
- `article_history` 表
- append-only 新闻归档表
- 或按天追加的历史新闻库

也就是说，Phase 3 当前应把这条边界写清楚：

```text
7d / 30d / 60d Trending 在第一版中基于当前快照做近似分析；
如需严格历史趋势统计，需单独建设历史新闻存储层。
```

### 7.4 轻量聚类算法说明
Phase 3 第一版不要求一上来做重型 embedding 聚类。  
至少要写清楚第一版用什么轻量聚类。

推荐第一版：
1. 基于标题归一化
   - 小写化
   - 去标点
   - 去公司名噪音
   - 去明显模板词
2. 结合 description 关键词重叠
3. 对同一主题表达做轻量相似归并
4. 再统计 cluster 在 `7d / 30d / 60d` 的出现数量

第一版可以接受的实现方式：
- 标题 token overlap
- 规则关键词归并
- 轻量相似度阈值

不要求第一版就上：
- embedding 全量聚类
- 全网事件图谱

### 7.5 `trend_cluster_key` 的稳定性约束
Phase 3 不能只说“有 trend cluster”，还要写清楚 key 的稳定性。

推荐要求：
- `trend_cluster_key` 由稳定规则生成
- 同一批趋势在短时间内刷新，不应频繁变 key
- key 应尽量和趋势核心表达绑定，而不是和数组顺序绑定

推荐格式：

```json
{
  "trend_cluster_key": "30d:data-center-demand"
}
```

规则要求：
- 包含窗口标签
- 包含归一化后的趋势短语
- 不使用临时随机值

### 7.5.1 `trend_cluster_key` 的 slug 归一化规则
为了避免不同程序员各写一套 slug 逻辑，Phase 3 第一版应明确：

```text
trend_cluster_key = {window_label}:{normalized_slug}
```

其中 `normalized_slug` 的生成规则固定为：

1. 小写化
2. 去掉标点
3. 统一空格和连字符
4. 去掉停用词和弱描述词，例如：
   - `the`
   - `a`
   - `an`
   - `remains`
   - `continues`
   - `strong`
   - `rising`
5. 统一常见主题表达：
   - `datacenter` -> `data-center`
   - `data center` -> `data-center`
   - `ai infrastructure` -> `ai-infrastructure`
6. 只保留 `2 ~ 4` 个核心主题词
7. 最终转成 `kebab-case`

示例：

```text
"Data Center Demand Remains Strong"
  -> "data-center-demand"

"Strong Data-Center Demand Continues"
  -> "data-center-demand"
```

最终：

```json
{
  "trend_cluster_key": "30d:data-center-demand"
}
```

这样可以保证：
- 同一趋势在短时间内刷新时 key 稳定
- 标题轻微变化不会导致 cluster key 漂移

### 7.6 推荐输出结构
Phase 3 的 `Trending` 更适合输出 **最多 6 条** 趋势，每条至少包括：

```json
{
  "trend_cluster_key": "30d:data-center-demand",
  "trend_title": "Data-center demand remains the clearest growth driver",
  "trend_summary": "Across the last 30 days, multiple EMS names repeatedly mentioned server demand, AI infrastructure build-out, and margin support tied to data-center programs.",
  "window_label": "30d",
  "supporting_count_7d": 4,
  "supporting_count_30d": 11,
  "supporting_count_60d": 18,
  "companies": ["FLEX", "JBL", "CLS"],
  "representative_items": [...]
}
```

### 7.7 LLM 的职责
Trending 可以使用 LLM，但 LLM 不负责先发现趋势，而负责：
- 为已形成的 cluster 生成 `trend_title`
- 生成 `trend_summary`
- 控制风格、长度和信息密度

### 7.7.1 Trend summary cache 的最小结构
如果 `Part 3` 要引入 trend summary cache，第一版至少应把结构写清楚，避免程序员只知道“要缓存”但不知道怎么存。

推荐最小结构：

```json
{
  "trend_cluster_key": "30d:data-center-demand",
  "summary": "...",
  "summary_version": "v1",
  "updated_at": "2026-04-18T10:30:00Z"
}
```

第一版规则：
- `trend_cluster_key` 作为主缓存 key
- 同一个 `trend_cluster_key` 在短时间内重复生成时，应优先命中缓存
- 如果 cluster 组成变化明显，或 `trend_cluster_key` 变化，则生成新缓存记录
- 第一版不要求复杂失效策略；只要求 key 稳定、可复用、可覆盖更新

### 7.8 为什么仍保留 6 条
Phase 3 仍可把前端展示上限保留为 6 条，但后端**不应强制凑满 6 条**。

更准确的规则应该是：

```text
后端动态发现趋势
  -> 返回 1 到 6 条真实趋势
前端最多展示 6 个位置
  -> 不足 6 条时，不补假趋势
```

也就是说：
- 如果本轮只发现 3 条真实趋势
  - 后端就只返回 3 条
- 如果本轮发现 8 条趋势
  - 后端按支持数量、时间窗口强度和公司覆盖度排序，只返回前 6 条

这 6 条不再是固定主题卡，而是：

```text
6 条最值得看的趋势总结
```

### 7.8.1 少于 6 条时的前端处理
如果后端返回少于 6 条趋势，前端不应为了布局完整去伪造趋势。

推荐做法：
1. 只渲染真实返回的 trend 卡片
2. 剩余位置不显示伪卡片
3. 如需维持网格整齐，可允许空白占位，但不展示虚构标题

也就是说：
- “最多 6 条”是展示上限
- 不是数据层必须凑满 6 条

这样可以避免两种坏结果：
- 趋势不够时硬塞低质量现象
- 趋势很多时又没有明确裁剪规则

## 八、跨来源重复合并升级

### 8.1 目标
Phase 3 需要在 `Phase 2` exact dedup 基线上，再增加一层：

```text
cross-source merge / event-level merge
```

目标：
- 尽量把同一条正式新闻或同一条 press release 的多站点重复合并成 1 条主记录
- 避免 `News Flash`、`Top News`、weekly summary 被同一事件重复污染

建议插入位置：

```text
normalize
  -> exact dedup (Phase 2)
  -> cross-source merge / event-level merge (Phase 3)
  -> ranking / top news / trending
```

### 8.2 `canonical_event_key` 生成规则
Phase 3 文档必须明确 `canonical_event_key` 至少第一版怎么生成。

推荐第一版：
1. 公司 ticker
2. 标题归一化短语
3. 事件模板标签
   - `earnings-call`
   - `press-release`
   - `official-announcement`
4. 条件性日期片段

示例：

```json
{
  "canonical_event_key": "flex:q4-fiscal-2026-earnings-call"
}
```

第一版不要求做到完美，只要求：
- 同一 press release 的不同转载站尽量得到同一 key
- key 生成规则稳定可复现

### 8.2.1 日期片段何时加、何时不加
`canonical_event_key` 里的日期片段不能随意拼接。  
第一版应明确：

1. **优先使用业务事件片段，不优先使用发布日期**
   - 如：
     - `q1 / q2 / q3 / q4`
     - `fiscal-2026`
     - `investor-day-2026`
     - `annual-meeting-2026`
   - 这类信息比网页发布日期更稳定

2. **如果标题或正文中已经有足够强的事件唯一性信息**
   - 不再额外加入精确到天的日期片段
   - 例如：
     - `flex:q4-fiscal-2026-earnings-call`
   - 这时同一 press release 被不同站在不同日期转载，仍应得到同一个 key

3. **只有在缺少业务事件片段、且标题过于泛化时，才引入时间片段**
   - 例如：
     - `Investor Day`
     - `Annual Meeting`
     - `Press Release`
   - 为了区分不同年份或不同批次事件，可加入较粗粒度时间片段：
     - `2026`
     - `2026-q2`
     - `2026-04`

4. **第一版不建议直接使用精确到天的发布日期**
   - 因为转载站可能晚一天发布
   - 这样会把同一事件误拆成多个 key

示例：

```text
"Flex Announces Date for Fourth Quarter and Fiscal 2026 Earnings Call"
  -> flex:q4-fiscal-2026-earnings-call
```

这里不再额外拼 `2026-04-15` 或 `2026-04-16`。

```text
"Flex Investor Day"
```

如果同类标题缺少季度或 fiscal 信息，可生成：

```text
flex:investor-day-2026
```

或：

```text
flex:investor-day-2026-q2
```

而不是直接使用精确发布日期。

### 8.3 主记录保留优先级
如果判断为同一正式新闻，Phase 3 第一版应采用可执行的保留规则。

第一版最小可执行规则：

```text
company_news_rss
  > Brave
```

也就是说：
- 如果 `company_news_rss` 与 Brave 指向同一正式新闻
  - 直接优先保留 `company_news_rss`
- 如果没有 `company_news_rss`
  - Brave 内部再做第二层选择

### 8.3.1 Brave 内部的第一版优先规则
Brave 内部不建议在 Phase 3 第一版靠“猜”来区分原始发布站和转载站。  
如果要做第二层优先，必须至少选择一种明确规则。

推荐第一版用 **域名白名单 + 路径模式**：

1. 原始发布源优先域名白名单
   - `prnewswire.com`
   - `businesswire.com`
   - `globenewswire.com`
   - `accessnewswire.com`
   - 已确认的公司官网 / IR 域名

2. 次级判断可用 URL 路径模式
   - `/news-releases/`
   - `/press-release/`
   - `/press-releases/`

3. 其余默认视为转载站 / 聚合站

Phase 3 第一版可以接受下面这条更保守的实现方式：

```text
company_news_rss > Brave
Brave 内部如果无法稳定判断原始发布站，
则只保留一个最佳 Brave 结果，不强行区分“原始发布站”与“转载站”。
```

这样程序员就不会陷入“文档要求区分，但没有判断标准”的状态。

### 8.4 合并后的最小结果
合并后前端仍只显示 1 条主新闻，但后端至少保留：
- `canonical_event_key`
- `matched_intents`
- `merged_sources`
- `merged_urls`
- `primary_source_type`

## 九、前端交互与 UI 约束

### 9.1 News Flash 固定显示
Phase 3 目标：
- `News Flash` 固定在当前主内容位置
- 不再允许横向滑动
- 只保留纵向滚动

### 9.2 右侧面板向右折叠
推荐增加 `News Flash / SEC Filings` 共用右侧面板的向右折叠能力。

推荐交互：
- 展开态：两列布局，右侧正常显示 News Flash / SEC Filings
- 折叠态：右侧面板完全隐藏（或缩成极窄的展开触发条），左侧 Top News + Trending 区域自动扩展占满全宽

关键要求：
- 折叠后，左侧主内容区（Top News、Trending）必须向右扩展，占满原右侧腾出的空间
- 不能只是把右侧缩成一个固定窄栏而左侧不变——那样没有意义
- 实现方式：在 grid 布局上切换列数，折叠时从 `grid-cols-[1.7fr_1fr]` 变成单列 `grid-cols-1`
- 折叠态可在右边缘保留一个极窄的展开按钮条（如 `w-8`），左侧内容区占 `1fr`

这件事的价值：
- 给 `Top News` 和 `Trending` 腾出空间
- 为未来加入图片卡片预留更大的主内容区

### 9.1.1 News Flash 横向滚动问题
News Flash 列表必须固定在右侧面板内，只允许纵向滚动。

常见错误原因：
- 列表项内部有宽度不受限的子元素（如长 URL、不换行文本）导致溢出
- 右侧面板没有设置 `overflow-x-hidden` 或 `min-w-0`
- 列表项的 grid/flex 子元素没有 `min-w-0`

修复要求：
- 右侧面板容器加 `overflow-x-hidden`
- 列表项内所有文本容器加 `min-w-0` 和 `truncate` 或 `break-words`

### 9.3 筛选区交互收口
在 Phase 2 已经确定关键词逻辑之后，Phase 3 负责把筛选区体验收口。

要求：
- preset 与手动输入框职责分开
- 点击 preset 时，不把内部规则塞进输入框
- 当前激活的 preset 与 search term 始终可见

### 9.4 关键词解释展示
点击某个 preset 后，前端仍应让用户看见当前实际使用的关键词。

例如：
- `Data Center`
  - `data center · server · rack · storage`
- `AI Infrastructure`
  - `artificial intelligence · ai server · gpu · accelerator · inference`

### 9.5 说明性提示图标与微文案
Phase 3 应补上少量说明性提示图标，目的不是增加功能，而是降低误解成本，让用户知道：
- 当前模块依赖哪些数据源或分析能力
- 哪些结果是辅助性的，不应被误认为精确证券终端
- 后续如果接入更高质量或付费数据源，哪些地方会变得更准确

这类提示图标不应打断主流程，建议统一使用：
- `?`
- `!`
- 或小灯泡图标

统一交互建议：
- 默认显示为小图标
- 点击后弹出简短 tooltip / popover
- 不要求 hover 数秒
- 文案尽量控制在 2~4 句内

#### 9.5.1 News Monitor 顶部提示图标
位置建议：
- 放在 `News Monitor` 标题区域附近
- 位于标题右侧或顶部 chip 行左端附近
- 不压住现有公司 chip

提示内容建议：

```text
当前新增公司功能以模糊搜索和公司识别为主，
支持常见上市公司名称、ticker、简称和部分中文别名。
由于暂未接入顶级证券主数据或付费公司资料 API，
当前识别结果属于辅助候选，后续接入更强数据源后准确率会继续提升。
```

这条提示的重点是：
- 说明当前是“可用的辅助识别”，不是完整证券主数据终端

#### 9.5.2 Top News 区域提示与微文案
Top News 区域应补两类说明：

1. `Top News` 标题附近的提示图标
   - 说明当前 `Top News` 依赖的能力边界
   - 建议文案：

```text
Top News 基于当前新闻流的排序结果，并结合摘要模型生成单条 summary 和 Weekly Summary。
当前总结主要依赖现有新闻元数据和摘要缓存，
后续若接入更强模型或更完整上下文，解释质量还会继续提升。
```

2. `Weekly Summary` 的浅灰色来源标注
   - 文字建议直接使用：

```text
Weekly
```

这里的作用不是表示“新闻来自周报源”，而是提示：
- 右侧模块是对当前 `Top News` 的周内整体总结
- 属于总结层，不是另一种新闻源

样式建议：
- 浅灰色
- 比主标题小一级
- 不与正文竞争视觉优先级

#### 9.5.3 Trending 的周期标注
`Trending` 区域应在标题右侧增加浅灰色时间周期提示。

推荐写法：
- `7d / 30d / 60d`
- 或更自然语言化的 `近 7 / 30 / 60 天`

不建议写成：
- `半年`

因为当前架构里并没有严格的 180 天历史趋势仓库；更准确的仍是：
- `7d`
- `30d`
- `60d`

这条标注的作用是：
- 让用户知道 trend 是时间窗口内的趋势总结
- 而不是固定主题筛选卡

#### 9.5.4 News Flash / SEC 区域提示图标
右侧 `News Flash / SEC Filings` 面板应增加一个说明性提示图标。

位置建议：
- 放在面板标题区域
- 位于 `News Flash / SEC Filings` tab 右侧
- 与数量标记或统计数字同一行，但不要遮挡切换按钮

提示内容建议：

```text
当前 News Flash 主要来自 Brave News API、公司新闻 RSS 和 SEC RSS。
其中 SEC 的部分 RSS 存在访问频率限制或返回不稳定的情况，
因此结果并不保证每次都完全一致。
如果未来接入更稳定的付费数据源或历史数据库，
News Flash 与 SEC 视图的完整度和稳定性会继续提升。
```

这条提示的重点是：
- 告知当前数据源组成
- 提前解释 SEC RSS 受限时为什么结果可能不稳定
- 说明未来增强方向
